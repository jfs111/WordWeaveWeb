# ✦🕸️ Word Weave Web

**Video Demo:** *(coming soon)*

**Description:**
Word Weave Web is a multi-tenant document intelligence platform that transforms unstructured PDF collections into navigable knowledge graphs. It combines semantic search (RAG) with graph-based relation detection to enable deep exploration of document corpora through natural language queries.

The system ingests PDF documents, extracts and chunks text, generates embeddings, performs automatic clustering, and uses a Large Language Model to detect semantic relations between content chunks. Users can then search, chat with, and visually explore their document knowledge graph through a web interface or a REST API designed for agent integration.

## Project Overview

The main idea behind Word Weave Web is to go beyond traditional Retrieval-Augmented Generation by enriching semantic search with **structural knowledge**. Where a standard RAG system retrieves the most similar text chunks to answer a question, Word Weave Web also traverses a knowledge graph of relations between chunks — discovering connections that pure vector similarity would miss.

Word Weave Web was designed for organizations managing large collections of pedagogical, technical, or regulatory documents. The initial use case involved **263 RNCP certification PDFs** (French national training standards), producing **1,205 chunks**, **38 thematic clusters**, and **2,846 inter-chunk relations** across 9 relation types.

The system is fully multi-tenant: each user can create multiple projects, each with its own isolated document collection, vector store, knowledge graph, and processing pipeline. Authentication is handled via JWT for the web interface and API keys for external agent integration.

## Features

**Document Ingestion Pipeline:**
- PDF upload with automatic text extraction (PyPDF2).
- Intelligent word-based text chunking with configurable size and overlap (default: 200 words / 50 words overlap, with fallback for short documents). Minimum chunk threshold: 10 words.
- Embedding generation using `paraphrase-multilingual-MiniLM-L12-v2` (384 dimensions, 50+ languages).
- Storage in Qdrant with per-project collection isolation (`proj_{project_id}`).
- PostgreSQL metadata tracking (documents, chunks, pages, word counts).

**Automatic Clustering:**
- Auto-K detection using a combined scoring of Elbow method (0.3), Silhouette score (0.4), and Davies-Bouldin index (0.3).
- K-means clustering on embedding vectors (`n_init=20`, `max_iter=300`).
- K range: `√(N/2)` to `2×√N` by steps of 2.

**LLM Relation Detection:**
- Pairwise analysis of chunks within each cluster using an LLM.
- 9 relation types: `PREREQUIS`, `COMPLEMENTAIRE`, `SIMILAIRE`, `METHODOLOGIQUE`, `APPLICATION`, `EXEMPLE`, `SUITE_LOGIQUE`, `TRANSVERSAL`, `AUTRE`.
- Confidence (0-1) and intensity scoring (FAIBLE/MOYENNE/FORTE) for each relation.
- Cosine similarity pre-filter and justification text generated by the LLM.
- Incremental commit strategy to preserve progress on large corpora.

**Semantic Search:**
- Vector similarity search via Qdrant.
- Graph enrichment: search results are augmented with related chunks discovered through Auto-Hop budget-based graph traversal (best-first, cosine similarity cost model). Benchmarked: +100% to +170% context chunks, +12% to +26% source relevance scores.

**Chat Graph-RAG:**
- Natural language question answering over the document corpus.
- Full pipeline: question → embed → search → graph enrichment → LLM generation → sourced answer.
- Markdown rendering of answers (tables, lists, code blocks) via marked.js.
- Clickable source references linking back to original document chunks.

**Interactive Knowledge Graph:**
- D3.js force-directed graph visualization.
- Nodes represent chunks, edges represent detected relations.
- Color-coded by cluster, filterable by relation type.
- Click on any node to open the document reader with full chunk content.

**Document Reader:**
- Overlay panel displaying chunk text, cluster assignment, and document metadata.
- LLM-powered reformatting: converts raw extracted text into clean Markdown with one click.
- Client-side caching of reformatted content to avoid redundant LLM calls.
- Related chunks displayed as colored tags by relation type.

**Document Analysis Pipeline:**
- Upload a new PDF and analyze it against the existing corpus.
- Automatic chunking, embedding, and cluster assignment (nearest centroid).
- Semantic search to find related existing chunks for each new chunk.
- LLM analysis of top matches with relation type, intensity, and justification.
- Generation of a full Markdown report with summary, cluster distribution, relation details, and reference documents.
- Downloadable `.md` report file.

**External REST API (for agents):**
- 10 endpoints with API key authentication (`X-API-Key`).
- Chat endpoint with customizable system prompt for agent integration.
- Auto-Hop graph traversal (budget-based best-first) with legacy BFS fallback for programmatic knowledge exploration. Benchmarked on two corpora (274 enterprise PDFs + 4 GDPR PDFs), Auto-Hop consistently improved retrieval: +125% domain coverage on cross-cutting questions, +50% sanction types discovered, and one critical procedure (PED) found exclusively through graph traversal. See `examples/benchmarks/` for full results.
- Cluster listing, graph data export, chunk retrieval.
- Compatible with LangChain, LlamaIndex, or any HTTP-based agent framework.

**Web Interface:**
- Dark-themed responsive UI ("Aurora Boréale" theme).
- Dashboard with project listing and statistics.
- Stepper-based project view: Upload → Clustering → Relations → Explorer.
- User authentication (register, login, JWT).
- Real-time progress tracking for long-running pipelines (clustering, relations, analysis).

## Architecture

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│   Service A      │     │   Service B       │     │    Service C         │
│   Storage :8000  │     │   Intelligence    │     │    Orchestrator      │
│                  │     │   :8001           │     │    :8002             │
│ • Qdrant CRUD    │     │ • Chunking        │     │ • Web UI (Stepper)   │
│ • Embeddings     │     │ • Embeddings      │     │ • Auth JWT           │
│ • Search         │     │ • Clustering      │     │ • Projects CRUD      │
│                  │     │   Auto-K          │     │ • Chat RAG           │
│                  │     │ • Relations LLM   │     │ • Document Analysis  │
│                  │     │                   │     │ • External API       │
└────────┬─────────┘     └─────────┬─────────┘     └──────────┬──────────┘
         │                         │                           │
   ┌─────┴─────────────────────────┴───────────────────────────┴─────┐
   │         PostgreSQL 15  •  Redis 7  •  Qdrant  •  LLM local    │
   └─────────────────────────────────────────────────────────────────┘
```

The system is divided into three services with clear responsibilities:

- **Service A (Storage):** Owns all persistent data — Qdrant vector collections and proxies PostgreSQL queries. No ML logic.
- **Service B (Intelligence):** Owns all compute-intensive ML operations — embedding generation, clustering, LLM calls. Stateless and horizontally scalable.
- **Service C (Orchestrator):** Owns all user-facing logic — authentication, web UI, API gateway, pipeline orchestration. Coordinates A and B.

This separation allows the ML engine to be scaled independently (e.g., GPU deployment for Service B) without affecting the web interface or storage layer.

## Design Decisions

### 1. Graph-RAG over Traditional RAG
Traditional RAG retrieves chunks based solely on vector similarity, which misses structural relationships between documents. By building a knowledge graph on top of the vector store, Word Weave Web can discover that Document A is a *prerequisite* for Document B, or that Document C *complements* Document D — information that pure cosine similarity cannot capture.

This architectural choice was inspired by research on knowledge-augmented retrieval and the observation that pedagogical documents have rich inter-dependencies (prerequisites, sequences, applications) that are lost in flat vector search.

### 2. Microservice Architecture — Three-Service Split
The three-service split ensures clear separation of concerns: storage (A), compute (B), and orchestration (C). This allows the ML engine to be scaled independently (e.g., GPU deployment for Service B) without affecting the web interface or storage layer.

### 3. Auto-K Clustering — No Manual Configuration Required
Users should not need ML expertise to use the system. The auto-K algorithm combines three complementary metrics with weighted averaging to automatically determine the optimal number of clusters. The formula `optimal_K = elbow_k × 0.3 + silhouette_k × 0.4 + davies_k × 0.3` was calibrated through experimentation on the RNCP corpus.

### 4. LLM-as-Judge for Relation Detection
Rather than relying on embedding similarity alone (which would only find "similar" content), the system uses an LLM to analyze chunk pairs and classify relations into 9 semantic types. This produces richer, more actionable knowledge graphs where relations carry meaning (prerequisite, complement, application, etc.) rather than just proximity scores.

The LLM is called via a local inference server (LM Studio), keeping all data on-premises and avoiding cloud API costs.

### 5. Dual Authentication — JWT + API Keys
The web interface uses JWT tokens for stateless session management, while the external API uses API keys for machine-to-machine authentication. This allows human users to interact through the browser while AI agents can programmatically access the same data.

### 6. Qdrant Per-Project Isolation
Each project gets its own Qdrant collection, named `proj_{project_id}`. This ensures complete data isolation between tenants — no cross-contamination of embeddings or search results, even if projects share the same Qdrant instance.

### 7. Background Jobs with Progress Tracking
Long-running operations (clustering, relation detection, document analysis) run as async background tasks with real-time progress updates stored in the PostgreSQL `jobs` table. The frontend polls job status every few seconds, providing users with step-by-step feedback rather than a blank loading screen.

### 8. Auto-Hop — Budget-Based Graph Traversal
Traditional graph traversal uses a fixed number of hops (e.g., 1-hop or 2-hop BFS), which is either too shallow (missing deep connections) or too broad (adding noise). The Auto-Hop algorithm solves this with a budget model: each traversal starts with a budget of 1.0, and each hop costs `1 - cosine_similarity`. High-quality paths (similarity > 0.8) can reach 5+ hops deep, while weak paths stop after 1-2 hops. This produces an adaptive exploration depth that matches the actual structure of the knowledge graph — no configuration needed from the user. The cosine similarity metric was chosen over LLM confidence to future-proof the system for a planned evolution toward fully vector-based relation qualification.

## Technologies Used

### Python & FastAPI — Backend Foundation
All three services are built with **FastAPI** for its async support, automatic OpenAPI documentation, and Pydantic validation. **PostgreSQL 15** handles relational metadata (users, projects, documents, chunks, relations, jobs) with **SQLAlchemy** as the async ORM.

### Qdrant — Vector Storage
**Qdrant** stores document embeddings with per-project collection isolation. It provides fast approximate nearest neighbor search with native cosine similarity scoring. Chosen over ChromaDB for its superior scalability, gRPC support, and production-grade reliability.

### Sentence-Transformers — Embedding Generation
The `paraphrase-multilingual-MiniLM-L12-v2` model generates 384-dimensional multilingual embeddings, chosen for its balance of quality, speed, and multilingual support (critical for the French-language RNCP corpus).

### Scikit-learn — Clustering
K-means clustering with auto-K detection uses **scikit-learn's** KMeans, silhouette_score, and davies_bouldin_score implementations.

### LM Studio — Local LLM Inference
All LLM operations (relation detection, chat, reformatting, document analysis) use a locally-hosted model via **LM Studio** with an OpenAI-compatible API. This keeps all data on-premises and eliminates cloud API costs. The system is model-agnostic — any OpenAI-compatible endpoint works.

### D3.js — Knowledge Graph Visualization
The interactive force-directed graph is rendered client-side with **D3.js**, providing zoom, pan, drag, tooltip, and click-to-read functionality.

### Docker — Containerized Deployment
The full stack runs in **6 Docker containers** orchestrated by docker-compose: PostgreSQL, Redis, Qdrant, Service A, Service B, Service C. Health checks ensure proper startup ordering.

### Redis — Caching Layer
**Redis** provides caching for frequently accessed data and serves as a message broker for future streaming features.

### Jinja2 + marked.js — Web Interface
Server-side HTML rendering with **Jinja2** templates, client-side Markdown rendering with **marked.js**, and vanilla JavaScript for all interactive features (tabs, chat, graph, document reader).

## How to Run

### Prerequisites
- **Docker** and **Docker Compose** installed
- **LM Studio** (or any OpenAI-compatible local LLM server) running on port 1234
- 16GB+ RAM recommended (8GB minimum)
- Windows / Linux / macOS

### Installation

**1. Clone the repository:**
```bash
git clone https://github.com/jfs111/WordWeaveWeb.git
cd WordWeaveWeb
```

**2. Configure environment:**
```bash
cp .env.example .env
# Edit .env with your own secrets if needed (defaults work for local dev)
```

**3. Start LM Studio:**
- Load a model (tested with `openai/gpt-oss-20b` and similar)
- Start the local server on `http://localhost:1234`

**4. Launch the full stack:**
```bash
docker-compose up -d --build
```

This starts 6 containers:

| Container | Service | Port |
|-----------|---------|------|
| `graphrag-postgres` | PostgreSQL 15 | 5432 |
| `graphrag-redis` | Redis 7 | 6379 |
| `graphrag-qdrant` | Qdrant | 6333 |
| `graphrag-storage` | Service A — Storage | 8000 |
| `graphrag-intelligence` | Service B — Intelligence | 8001 |
| `graphrag-orchestrator` | Service C — Web UI & API | 8002 |

**5. Access the application:**
- Open your browser to **http://localhost:8002**
- Create an account and start your first project

### Quick Start Workflow
1. **Register** an account
2. **Create a project** from the dashboard
3. **Upload PDFs** — automatic text extraction, chunking, and embedding
4. **Run clustering** — automatic K detection groups your chunks into themes
5. **Run relation detection** — LLM analyzes pairs within each cluster
6. **Search** your corpus semantically
7. **Chat** with your documents via Graph-RAG
8. **Explore** the knowledge graph visually
9. **Analyze** new documents against your corpus

### Configuration

| Variable | Description | Default |
|----------|-------------|---------|
| `LLM_URL` | LLM API URL (OpenAI-compatible) | `http://host.docker.internal:1234/v1` |
| `LLM_MODEL` | Model name | `openai/gpt-oss-20b` |
| `EMBEDDING_MODEL` | Embedding model | `paraphrase-multilingual-MiniLM-L12-v2` |
| `POSTGRES_PASSWORD` | PostgreSQL password | *(see .env)* |
| `JWT_SECRET` | JWT signing key | *(see .env)* |

The LLM is used by **Service B** (relation detection) and **Service C** (Chat RAG, document analysis).
To change the model, edit `docker-compose.yml` then `docker-compose up -d` (no rebuild needed).

## Unit Tests

The project includes a suite of **115 unit tests** covering the core business logic of all three services. Tests run entirely without Docker — no database, no LLM, no Qdrant required. All external dependencies (PostgreSQL, OpenAI client) are mocked.

### Prerequisites

- **Python 3.10+** — [download here](https://www.python.org/downloads/) — check **"Add Python to PATH"** during installation
- No other prerequisites: the test runner creates and configures its own isolated virtual environment automatically

### Running the Tests

**Windows (PowerShell):**
```powershell
powershell -ExecutionPolicy Bypass -File .\run_tests.ps1
```

> If your system blocks `.ps1` scripts by default (common on freshly installed Windows), the `-ExecutionPolicy Bypass` flag overrides this restriction for the duration of this single command — no system-wide setting is changed.

**Linux / macOS:**
```bash
bash run_tests.sh
```

Both scripts automatically create a `.venv-tests` virtual environment, install all test dependencies from `requirements-test.txt`, and launch pytest.

### Running a Specific Test File

Once the virtual environment is active (`.venv-tests\Scripts\activate.bat` on Windows or `source .venv-tests/bin/activate` on Linux/macOS), you can target a single file or test class:

```bash
# Run only the chunker tests
python -m pytest tests/service_b/app/services/test_chunker.py -v

# Run only a specific class
python -m pytest tests/service_c/app/services/test_auth_service.py::TestDecodeToken -v

# Run with detailed failure output
python -m pytest tests/ -v --tb=long
```

### Known Behaviors Documented by Tests

Two edge cases in the source code are explicitly documented through failing-by-design tests:

- **`TextChunker`** — passing `chunk_overlap >= chunk_size` raises a `ValueError` (step becomes zero in `range()`). This is expected behavior: the chunker does not silently ignore an invalid configuration.
- **`hash_password`** — passwords longer than 72 bytes raise a `ValueError`. This is a bcrypt library limitation; the test `test_password_over_72_bytes_raises` documents it so contributors are aware.

### Notes on Warnings

Running the auth tests produces JWT `InsecureKeyLengthWarning` messages. These are expected: the test suite uses a short dummy secret (`"change-me-in-production"`). They do not affect test results. In production, set `JWT_SECRET` to a value of at least 32 characters in your `.env` file.

## Models & Algorithms

### Embedding
- Model: `paraphrase-multilingual-MiniLM-L12-v2` (384 dimensions)
- Multilingual (50+ languages)

### Clustering Auto-K
- Algorithm: KMeans with automatic optimal K search
- K range: `√(N/2)` to `2×√N` by steps of 2
- 3 combined metrics: Elbow (0.3) + Silhouette (0.4) + Davies-Bouldin (0.3)
- `n_init=20`, `max_iter=300`, `random_state=42`

### Relation Detection
9 relation types detected by LLM:

| Type | Description |
|------|-------------|
| PREREQUIS | A must be understood before B |
| COMPLEMENTAIRE | A and B enrich each other |
| SIMILAIRE | A and B cover the same topic |
| METHODOLOGIQUE | Methodological link |
| APPLICATION | A is a practical application of B |
| EXEMPLE | A illustrates B with an example |
| SUITE_LOGIQUE | A is the logical continuation of B |
| TRANSVERSAL | Cross-domain thematic link |
| AUTRE | Uncategorized relation |

Each relation has: intensity (FAIBLE/MOYENNE/FORTE), confidence (0-1), cosine similarity, LLM justification.

### Auto-Hop Graph Traversal
Instead of a fixed number of hops, the Auto-Hop algorithm uses a **budget-based best-first** strategy to adaptively explore the knowledge graph.

- **Budget model**: Starting budget of 1.0. Each hop costs `1 - cosine_similarity`. A high-similarity relation (0.95) costs only 0.05, enabling deep traversal. A low-similarity relation (0.30) costs 0.70, stopping early.
- **Strategy**: Best-first (greedy) — always pick the most similar unvisited neighbor.
- **Deduplication**: visited set ensures each node is explored only once.
- **Safety**: hard `max_hops` limit (default 10) prevents runaway traversal.
- **Dual mode**: The API supports both `"auto"` (budget-based) and `"fixed"` (legacy BFS) modes.

Typical behavior on the NovaTech corpus (274 PDFs, 2,406 relations): 3-8 hops per starting chunk (up to 10 on high-similarity document series), budget consumption of 0.73-0.88, traversing all 9 relation types with COMPLEMENTAIRE and SIMILAIRE dominant. The Chat RAG pipeline runs Auto-Hop from each vector search result, typically producing 4-8 vector + 17-20 graph = 20-27 context chunks. On the RNCP corpus (263 PDFs, 3,499 relations): 3-4 hops, 7 vector + 15 graph = 22 chunks.

## Benchmarks

Two corpora were tested to evaluate the impact of Graph-RAG Auto-Hop versus classical vector-only RAG. Each question was asked twice — once without graph enrichment (vector search only), once with Auto-Hop enabled — and the results were compared on coverage, source quality, and response depth.

Full benchmark details with per-question logs and Auto-Hop traversal traces are available in `examples/benchmarks/`.

### Corpus 1 — NovaTech Solutions (Enterprise Documentation)

| Parameter | Value |
|-----------|-------|
| **Documents** | 274 PDFs (contracts, amendments, policies, CSE minutes, incident reports, job descriptions) |
| **Chunks** | 395 |
| **Clusters** | 32 |
| **Relations** | 2,406 |
| **LLM** | openai/gpt-oss-20b (local, LM Studio) |

**5 questions** testing amendment chains (CDI → avenant → avenant), cross-domain references (HR → IT → Security → Legal), and document discovery:

| # | Question | RAG Classic | Graph-RAG Auto-Hop | Key Gain |
|---|----------|------------|-------------------|----------|
| Q1 | Employee confidentiality obligations | 6 obligations, max 62% | 5 hierarchical levels with differentiated durations, max 82% | +IP obligations, +incident reporting, 2yr vs 5yr durations |
| Q2 | Telework rules evolution | 5 steps, max 64% | 4 enriched steps, max 90% | +Contractual integration (AV-CDI-2024 art. 9) |
| Q3 | Security incident procedure | 9 steps, max 68% | 7 steps + archiving, max 80% | +RCD archiving (5 years) |
| Q4 | AI generative usage rules | 4 categories, max 73% | 3 categories, max 90% | +17% source scores, more relevant context |
| Q5 | Right to erasure procedure | 6 generic steps, max 57% | 7 precise steps with §§ references, max 82% | **PED unfindable by vector search**, discovered via graph |

**Q5 is the most demonstrative case:** vector search scored only 49-57% (wrong documents returned), while Auto-Hop followed COMPLEMENTAIRE and TRANSVERSAL relations to discover the Rights Exercise Procedure (PED) — producing a precise 7-step procedure with DPO email, 48h acknowledgment, CNIL identity verification, and CCT/PTR references.

### Corpus 2 — RGPD Regulatory Documents

| Parameter | Value |
|-----------|-------|
| **Documents** | 4 PDFs (GDPR obligations, personal data rights, EU AI Act) |
| **Chunks** | 415 |
| **Clusters** | 24 |
| **Relations** | 2,390 |

**3 questions** testing Graph-RAG on a homogeneous regulatory corpus:

| # | Question | RAG Classic | Graph-RAG Auto-Hop | Key Gain |
|---|----------|------------|-------------------|----------|
| Q1 | Minimum obligations for digital solutions | 4 domains | 9 domains | **+125% coverage** (+transparency, rights, responsibility, subcontracting, incidents) |
| Q2 | Data subject rights | 8 rights | 9 rights | +consent withdrawal, enriched practical details |
| Q3 | GDPR non-compliance sanctions | 4 sanction types | 6 sanction types | +50% types, TRANSVERSAL relations dominant |

### Key Findings

1. **Graph-RAG improves all question types**, from +12% (focused questions) to +125% (cross-cutting questions) in coverage.

2. **The gain is proportional to question complexity:**
   - Cross-cutting questions (Q1 RGPD, Q1/Q5 NovaTech): major gains — new documents discovered
   - Amendment chain questions (Q2 NovaTech): SUITE_LOGIQUE relations reconstruct contractual history
   - Focused questions (Q2 RGPD, Q4 NovaTech): qualitative gains — higher source scores, richer details

3. **Relation types match question semantics:**
   - Confidentiality → SIMILAIRE (CDI series traversal)
   - Telework → SUITE_LOGIQUE + APPLICATION (amendment chains)
   - Security → COMPLEMENTAIRE (procedure chains)
   - Rights → TRANSVERSAL (cross-document links)

4. **Source similarity scores are systematically higher** with Graph-RAG (+12% to +26%), indicating more relevant context for the LLM even when thematic coverage is equivalent.

5. **Auto-Hop budget of 1.0 is well calibrated:** average consumption 0.73-0.88, with traversals reaching 10 hops on high-similarity document series (CDI contracts: 0.97-0.99 similarity) while stopping at 3-4 hops on weaker connections.

6. **Limitation identified:** Graph-RAG starts from vector search results. If vector search misses a document entirely (RGPD Q3: sanctions in `vosdroits` not found), the graph cannot compensate. A future **hierarchical super-index** (1 embedding per document for document-level pre-filtering) would address this.

> **Reproduce these results:** Upload the corpora from `examples/corpus_novatech/` and `examples/corpus_rgpd/`, run the full pipeline, and test the questions listed in the benchmark files.

## Challenges Encountered

### 1. Validating the Graph-RAG Concept
The initial question was whether LLM-detected relations would actually improve retrieval quality over pure vector search. A proof of concept using the RNCP corpus compared standard RAG (top-5 vector search) with Graph-RAG (top-5 + Auto-Hop graph neighbors). The graph-enriched approach consistently surfaced relevant context that vector search alone missed, particularly for questions spanning multiple topics. With the Auto-Hop algorithm (budget-based best-first traversal using cosine similarity as cost), the system adaptively explores 2-5 hops deep when relations are strong, while stopping early on weak connections — producing richer context without noise. This validated the core architecture. This was later confirmed by formal benchmarks on two additional corpora (NovaTech enterprise docs and RGPD regulatory PDFs), where Graph-RAG with Auto-Hop consistently outperformed classical RAG across all 8 test questions — see the Benchmarks section below.

### 2. Scaling LLM Relation Detection
With 1,205 chunks, naive pairwise comparison would require ~725,000 LLM calls — completely impractical. The solution was to first cluster chunks into thematic groups, only compare within clusters, use embedding similarity as a pre-filter (cosine ≥ 0.6), and implement incremental commits. This reduced LLM calls from ~725K to ~4,500 while maintaining coverage of meaningful relations.

### 3. Multi-Service Communication Reliability
Orchestrating three FastAPI services communicating via HTTP introduced challenges around timeout management (embedding generation can be slow on CPU), error propagation across service boundaries, database session management in async background tasks, and Docker networking. FastAPI's `BackgroundTasks` silently swallowed exceptions, requiring a switch to `asyncio.create_task` with explicit error logging.

### 4. ChromaDB to Qdrant Migration
The initial implementation used ChromaDB for vector storage. ChromaDB's `collection.get()` crashed with `list assignment index out of range` on collections with >500 items, and L2 distance scores required manual conversion (`1 / (1 + distance)`). The migration to Qdrant resolved these issues — Qdrant handles large collections natively, returns cosine similarity directly, and supports batch scroll operations for reliable data retrieval.

### 5. Frontend Without a Framework
The entire web interface was built with vanilla JavaScript, Jinja2 templates, and CSS — no React, no Vue, no build step. This was a deliberate choice to minimize complexity and dependencies, but it required careful management of state (tabs, chat history, graph data, analysis jobs) across a single-page-like experience.

### 6. Using AI Tools Responsibly
Claude AI assisted in generating significant portions of the codebase. However, every generated component required careful review and architectural adaptation, debugging of subtle issues, integration testing across all three services, and ensuring consistency between services that share models and schemas. AI-assisted development requires strong engineering oversight — the AI accelerates implementation but cannot replace architectural understanding.

## Future Improvements

- **Agent Framework Integration:** LangGraph agents for autonomous multi-step research across the knowledge graph.
- **GPU Acceleration:** Deploy Service B with NVIDIA Container Toolkit for faster embedding generation and larger local LLMs.
- **Streaming Chat (SSE):** Server-Sent Events for real-time token streaming.
- **Auto-Hop Visualization:** Interactive path explorer highlighting Auto-Hop traversal paths in the D3.js graph (budget consumption, hop cost per edge).
- **Vector-Based Relation Qualification:** Replace LLM-based pairwise relation detection with vector operations and clustering on embedding differences, using the LLM only for sampling-based cluster label qualification. This would dramatically reduce LLM calls while maintaining relation quality.
- **Graph Export Formats:** GraphML, GEXF, and JSON-LD export for integration with Neo4j, Gephi, NetworkX.
- **Advanced Clustering:** Hierarchical clustering and LLM-generated cluster topic labels.
- **Prometheus + Grafana:** Observability with metrics dashboards for pipeline performance and LLM latency.
- **Rate Limiting:** Redis-based rate limiting on the external API.
- **Multi-Format Ingestion:** DOCX, Markdown, HTML, and plain text support alongside PDF.
